Первым документом работы с проектом является "Анализ ТЗ"

Система помогает спланировать поездку. Для этого люди хранят информацию, такую как билеты, отели, трансфер, активность. А если люди едут группами, то каждый имеет свое мнение из-за чего есть необходимость в одной системе для всех, а также чтоб в нем были все сервисы из перечисленных. 

При прочтении документа первый раз присутствует вопрос о "Синхронизации расписания и уведомлений", "Обратная связь по поездке". 

Отсюда вытаскиваем сущности системы: Общая информация(Trips), информация о участникках (Trip_Participants), выбор билетов (Tiсkets), выбор отеля (Hotel), интересные места (Interesting_place), публикации по вкусу (Recommendations), уведомления (Notifications), Персональные данные (personal).

Учесть:
1. Многопользовательский режим
2. Интеграции с внешним API
3. Обработка платежей, паспортные данные
4. Работа с тайм-зонами и международными форматами данных
5. Конфликты расписаний и пересадок
6. Персонализация без утечки данных
7. Синхронизации при плохом интернете

"Работа с тайм-зонами и международными форматами данных" решение -- Все события в БД сохраняются в UTC. Клиентское приложение отображает время в часовом поясе объекта (аэропорта/отеля)
"Многопользовательский режим" -- Если два пользователя меняют один объект, система блокирует автоматическое сохранение и просит подтвердить финальную версию
"Персонализация без утечки данных" -- Паспортные данные шифруются (AES-256) и передаются внешним API через защищенный шлюз, не сохраняясь в логах рекомендательной системы
"Обработка платежей, паспортные данные" -- Пользователь вводит данные в защищенном фрейме провайдера, а нам возвращается только статус «Успешно» и token_id для транзакции. Хранятся в отдельном зашифрованном микросервисе. Доступ к ним имеет только модуль бронирования в момент покупки билета.
"Конфликты расписаний и пересадок" -- При добавлении нового элемента (например, трансфера) система проверяет время окончания предыдущего (рейса) и время начала нового.
"Синхронизации при плохом интернете" -- Все действия пользователя в оффлайне записываются в локальную базу браузера/телефона. Как только телефон ловит Edge/Wi-Fi, приложение начинает по очереди отправлять накопленные действия на сервер. Если за время оффлайна кто-то другой уже изменил этот же отель, сервер присылает ответ: «Ошибка 409 (Conflict)». Приложение открывает окно: «Пока вы были не в сети, планы изменились. Оставить вашу версию или обновить?».
"Интеграции с внешним API" -- Для получения данных об отелях и рейсах система интегрируется с внешними поставщиками (Booking, Amadeus) через API Gateway.
